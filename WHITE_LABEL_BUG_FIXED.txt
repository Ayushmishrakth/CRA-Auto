================================================================================
                            THE BUG EXPLAINED
================================================================================

SYMPTOM:
  "Network Error x15" when clicking "Generate & Download"
  Report never generates
  File never downloads
  Retries happen 15 times

ROOT CAUSE:
  Manual Content-Type header broke Axios's multipart form-data encoding

  BROKEN CODE:
    headers: {
      "Content-Type": "multipart/form-data",  // ❌ This breaks it
    }

  PROBLEM:
    - Axios needs to add a boundary marker when sending FormData
    - The boundary looks like: "boundary=----WebKitFormBoundaryXXXXXX"
    - Without the boundary, the backend can't parse the multipart request
    - Backend returns 422 (Unprocessable Entity)
    - Frontend treats this as "Network Error" and retries

HOW IT FAILS:
  1. Frontend sends POST /reports/assessments/{id}/generate
  2. Content-Type header is just "multipart/form-data" (no boundary)
  3. Request body has no boundary markers
  4. Backend's FastAPI multipart parser fails
  5. Returns 422 error
  6. Frontend retries 15 times
  7. User sees "Network Error x15"

CHANGE 1: reportApi.js - Remove Broken Header

  BEFORE (Line 25-31):
    ❌ headers: {
    ❌   "Content-Type": "multipart/form-data",
    ❌ },

  AFTER (Lines 25-28):
    ✅ // Do NOT manually set Content-Type header
    ✅ // When Axios detects FormData, it automatically sets:
    ✅ // "Content-Type": "multipart/form-data; boundary=..."
    ✅ // If you manually set the header, Axios won't add the boundary

  WHY THIS WORKS:
    - Axios detects FormData automatically
    - Axios generates proper boundary marker
    - Backend can parse multipart correctly
    - No more 422 errors

CHANGE 2: reportApi.js - Add Error Logging

  BEFORE (Line 32-40):
    const response = await api.post(...);

  AFTER (Lines 22-40):
    try {
      const response = await api.post(...);
      // ... normal code ...
    } catch (error) {
      console.error("[REPORT API] generateCustomizedReport failed", {
        assessmentId,
        format,
        companyName,
        hasLogo: !!logoFile,
        errorMessage: error.message,
        errorStatus: error.response?.status,
        errorData: error.response?.data,
      });
      throw new Error(...);
    }

  WHY THIS HELPS:
    - Developers can see actual error in console
    - Shows which field caused the problem
    - Includes HTTP status and server response
    - Much better than generic "Network Error"

CHANGE 3: reports.py - Add Request Logging

  BEFORE (Line 73-76):
    logger.info(f"[REPORT] Starting generation for {assessment_id}")
    logger.info(f"[REPORT] Custom: company={company_name}...")

  AFTER (Lines 73-100):
    logger.info(
        "[REPORT] Starting generation for assessment=%s, user=%s, format=%s",
        assessment_id,
        current_user.id,
        report_format,
    )
    logger.info(
        "[REPORT] Parameters: company_name=%s (len=%d), address=%s...",
        bool(company_name),
        len(company_name) if company_name else 0,
        bool(company_address),
        ...
    )
    if logo:
      logger.info(
          "[REPORT] Logo file: filename=%s, content_type=%s, size=%s bytes",
          logo.filename,
          logo.content_type,
          len(await logo.read()) if logo else "unknown",
      )

  WHY THIS HELPS:
    - Structured logging shows what request received
    - Easier to debug multipart issues
    - Can see file size and type
    - Helps diagnose upload problems

CHANGE 4: reports.py - Better Exception Handling

  BEFORE (Line 234-237):
    except HTTPException:
      raise
    except Exception as exc:
      logger.exception(f"[REPORT] Generation failed: {exc}")
      raise HTTPException(status_code=500, detail=f"Report generation failed...")

  AFTER (Line 235-252):
    except ValueError as ve:
      logger.error("[REPORT] Validation error: %s", str(ve), exc_info=True)
      raise HTTPException(status_code=400, detail=str(ve))
    except HTTPException:
      raise
    except Exception as exc:
      logger.exception(
          "[REPORT] Unexpected error during generation for assessment=%s: %s",
          assessment_id,
          exc,
          exc_info=True,
      )
      raise HTTPException(
          status_code=500,
          detail=f"Report generation failed: {str(exc)}"
      )

  WHY THIS HELPS:
    - ValueError gets 400 status (client error)
    - Other errors get 500 status (server error)
    - Assessment ID logged for troubleshooting
    - Full traceback logged for developers
